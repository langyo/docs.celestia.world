# Предварительные требования TIA Portal — подключение evernight

> **Цель**: один раз подготовить ПЛК Siemens S7-1200/1500 в TIA Portal, чтобы
> evernight мог подключаться, самоорганизовываться и читать/записывать
> устройство **без дальнейшего вмешательства человека**. Это разовая настройка
> свойств CPU — ваша логика программы (лесенка/SCL) не затрагивается.

evernight говорит с ПЛК Siemens по **двум каналам**. Выберите по тому, что
доступно на вашем ПЛК:

| Канал | Порт | Стиль доступа | Подготовка TIA | Рекомендуется для |
|-------|------|---------------|----------------|-------------------|
| **S7comm** | 102 | R/W сырых байтов M / I / Q / DB | PUT/GET + неоптимизированные DB | legacy, лёгкий, без лицензии OPC UA |
| **OPC UA** | 4840 | символьный, самоописывающий | включить встроенный сервер | **рекомендуется** — автообнаружение, оптимизированные DB OK |

Если можете включить OPC UA, предпочитайте его: evernight **обходит** всё
символьное адресное пространство автоматически, ручной ввод символов отсутствует.

---

## Путь A — S7comm (доступ к сырым регистрам)

### A.1 Включить связь PUT/GET

S7-1200/1500 по умолчанию блокируют внешнее чтение/запись S7.

1. Откройте проект в **TIA Portal**.
2. В представлении устройств/сети **кликните CPU**.
3. Свойства → **Защита и безопасность (Protection & Security) → Механизмы соединения (Connection mechanisms)**.
4. Отметьте **«Разрешить доступ через связь PUT/GET от удалённого партнёра (Permit access with PUT/GET communication from remote partner)»**.
5. Загрузите конфигурацию оборудования в CPU.

### A.2 Сделать целевые DB неоптимизированными

Оптимизированный доступ к блокам (по умолчанию на S7-1200/1500) не имеет
фиксированного байтового смещения, поэтому чтение по абсолютному адресу
завершается ошибкой. Для каждого DB, который evernight должен читать/писать:

1. ПКМ по DB → **Свойства (Properties)**.
2. Снимите флаг **«Оптимизированный доступ к блокам (Optimized block access)»**.
3. Перекомпилируйте и загрузите.

> Маркеры M и образы процесса I/Q всегда адресуются по байту — изменений не
> требуется. Этот шаг касается только DB.

### A.3 Подключение из evernight

```
s7://192.168.1.10:102?rack=0&slot=1
```

- S7-1200/1500: `rack=0, slot=1`
- S7-300: `slot=2`

Самоорганизация в коде:

```rust
use evernight::protocol::auto_provision;

let profile = auto_provision("192.168.1.10").await?;
// profile.data_blocks / profile.db_structures теперь описывают каждый читаемый DB
```

## Путь B — OPC UA (рекомендуется)

### B.1 Требования к прошивке и лицензии

- Прошивка CPU **V2.0+** (V2.5+ для методов OPC UA).
- **Лицензия времени выполнения SIMATIC OPC UA** по классу CPU (назначается в
  Свойства CPU → **Лицензии времени выполнения (Runtime licenses) → OPC UA**).
  Требуется для соответствия.

### B.2 Включить сервер OPC UA

1. **Кликните CPU** в представлении сети/устройства.
2. Свойства → **OPC UA → Общие (General)**: введите имя сервера.
3. Свойства → **OPC UA → Сервер (Server)**: отметьте **«Активировать сервер OPC UA (Activate OPC UA server)»**.
4. Назначьте сервер **интерфейсу PROFINET**, к которому подключится клиент.

### B.3 Открыть символьные переменные

- **OPC UA → Сервер → Интерфейс сервера (Server interface)**: выберите
  **«Стандартный интерфейс сервера SIMATIC (Standard SIMATIC server interface)»**,
  чтобы каждая символьная переменная/DB (включая оптимизированные DB)
  публиковалась автоматически.

### B.4 Аутентификация и безопасность

- **OPC UA → Сервер → Аутентификация пользователя (User authentication)**:
  анонимно (для доверенной LAN) или имя/пароль.
- **OPC UA → Сервер → Безопасность (Security)**: выберите политику. `None`
  проще всего для первого соединения; `Sign & Encrypt` для продакшена.
- На прошивке **V3.1+ с TIA V19+** предоставьте функциональную роль / право
  времени выполнения **«доступ к серверу OPC UA (OPC UA server access)»**
  подключающемуся пользователю.

### B.5 Доверять сертификату клиента

Клиенты OPC UA представляют сертификат X.509 при подключении; ПЛК помещает
неизвестные сертификаты в карантин. После первой попытки evernight:

1. TIA Portal → **CPU → Сертификаты (Certificates)** (онлайн), **или**
2. **Веб-сервер** ПЛК → «Связь с клиентами OPC UA», **или**
3. менеджер сертификатов на дисплее CPU.

Затем **принять/доверить** сертификат клиента evernight.

### B.6 (опционально) Экспорт XML NodeSet OPC UA

Файл NodeSet — офлайн-карта всех переменных, полезен для планирования без живого
соединения:

1. Свойства CPU → **OPC UA → Сервер → Экспорт (Export)**.
2. Кликните **«Экспортировать XML-файл OPC UA (Export OPC UA XML file)»**,
   сохраните `*.Opc.Ua.NodeSet2.xml`.

### B.7 Загрузка

Загрузите **конфигурацию оборудования**. Это свойства CPU, а не программная
логика — ваш код лесенки/SCL остаётся нетронутым.

### B.8 Подключение из evernight

URL конечной точки:

```
opc.tcp://192.168.1.10:4840
```

evernight подключается как клиент OPC UA, **обходит** всё символьное дерево и
читает/пишет по имени — без ручного ввода символов, оптимизированные DB
включительно.

## Проверка связности (нулевые риски)

Прежде чем управлять выходами, подтвердите живость каналов зондами только для
чтения:

```bash
# Кто-то говорит S7comm на порту 102?
evernight probe 192.168.1.10 --ports 102

# Поднят ли сервер OPC UA на 4840?
evernight probe 192.168.1.10 --ports 4840
```

Оба — пассивные рукопожатия — ничего не читают и не пишут.

## Границы безопасности

- **Никогда** не помещайте блокировки безопасности (аварийные остановы, концевики,
  перегрузку) на путь S7/OPC UA. Держите их в скане ПЛК. Обрыв сети не должен
  отключать функцию безопасности.
- Управление evernight подходит для **медленных** нагрузок (клапаны, конечные
  автоматы, переключение режимов). Время кругового хода S7/OPC UA ~10–50 мс —
  достаточно для надзора, слишком медленно для движения/серво.
- Предпочитайте запись **командных битов M**, на которые реагирует существующая
  логика ПЛК (вы перехватываете источник триггера), а не прямую запись выходов Q.

## Устранение неисправностей

| Симптом | Вероятная причина | Исправление |
|---------|-------------------|-------------|
| Соединение S7 отклонено / нет COTP-подтверждения | PUT/GET не включён; неверные rack/slot; брандмауэр | A.1; проверьте `rack=0 slot=1` (1200/1500) |
| Чтение DB возвращает «optimized» / InvalidAddress | Включён оптимизированный доступ к блокам | A.2 — снять оптимизацию, перекомпилировать |
| Конечная точка OPC UA недостижима | Сервер не активирован; не загружен; нет лицензии | B.2 / B.7 / B.1 |
| OPC UA подключается, затем отклоняется | Сертификат клиента не доверен | B.5 |
| Обход возвращает пусто | Стандартный интерфейс SIMATIC не включён | B.3 |
