# Архитектура feature-флагов

- **Статус**: Принято
- **Дата**: 2026-06-09
- **Авторы**: Evernight Core Team

## Контекст

Монолитный граф зависимостей Evernight вынуждает всех потребителей подтягивать каждую зависимость (webrtc, russh, screenshots, sysinfo), даже если им нужен только SSH или аппаратная телеметрия. Это увеличивает время компиляции и размер бинарных файлов для downstream-пользователей.

## Решение

Использовать feature-флаги Cargo для разделения крейта на следующие функциональные возможности:

| Feature      | Ограничивает                                  |
|--------------|----------------------------------------------|
| `screen`     | Модуль захвата экрана + крейт `screenshots`  |
| `webrtc`     | Модуль WebRTC + крейт `webrtc` (подразумевает `screen`) |
| `remote-ssh` | Модуль обработчика SSH + крейт `russh`        |
| `hardware`   | Модуль аппаратной телеметрии + крейт `sysinfo` |
| `protocol`   | Типы протоколов/сообщений (без тяжёлых зависимостей) |
| `tunnel`     | Модуль TCP-туннелей                           |
| `full`       | Все возможности (по умолчанию)                 |

Каждый feature-флаг ограничивает как модуль, так и его зависимости. Feature `webrtc` подразумевает `screen`, так как сеансы WebRTC требуют захвата экрана.

## Последствия

### Положительные

- Потребители компилируют только то, что им нужно
- Сокращение времени компиляции при частичном использовании

### Отрицательные

- Матрица feature-флагов растёт; необходимо тестировать каждую комбинацию в CI
