# Разделение модулей и владение типами

- **Статус**: Принято
- **Дата**: 2026-06-09
- **Авторы**: Evernight Core Team

## Контекст

Исходная кодовая база размещала все типы данных в едином монолите `types.rs` — смешивая типы screen, hardware, remote, protocol и tunnel. Это создавало риски циклических зависимостей и делало неясным, какой модуль «владеет» какими типами. Добавление новой функциональности (например, VNC) требовало бы изменения общего файла типов.

## Решение

Переместить типы каждого домена в собственный файл `types.rs` модуля (например, `screen/types.rs`, `hardware/types.rs`). Корневые `types.rs` и `prelude.rs` реэкспортируют все типы для обратной совместимости. Каждый модуль является единственным владельцем своих типов и может развиваться независимо.

## Последствия

### Положительные

- Ясное владение типами; отсутствие межмодульных конфликтов типов
- Новые модули не затрагивают существующие файлы

### Отрицательные

- Потребители должны импортировать из правильного модуля или использовать prelude
- Слой реэкспорта добавляет косвенность
