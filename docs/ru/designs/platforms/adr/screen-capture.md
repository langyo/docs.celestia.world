# Архитектура захвата экрана

- **Статус**: Принято
- **Дата**: 2025-06-09
- **Авторы**: Evernight Core Team

## Контекст

Evernight требуется кроссплатформенная подсистема захвата экрана, подающая кадры в конвейер потоковой передачи WebRTC. Подсистема должна работать на Windows, macOS и Linux с минимальной задержкой и поддерживать как аппаратно-ускоренные, так и программные пути захвата.

Ключевые ограничения:

- **Бюджет задержки**: сквозной путь захват-кодирование должен укладываться в 16 мс при 60 FPS
- **Zero-copy где возможно**: кадры должны достигать кодировщика без лишних копий
- **Поддержка горячего подключения**: дисплеи и GPU могут появляться или исчезать во время работы
- **Обработка разрешений**: macOS требует разрешения на запись экрана; Linux требует доступа к протоколу XDG/X11 или Wayland

## Решение

Мы принимаем **бэкенд захвата на основе типажа** с платформенно-специфичными реализациями, выбираемыми во время компиляции через Cargo-функции. Каждый бэкенд реализует типаж `FrameProvider`:

```rust
#[async_trait]
pub trait FrameProvider: Send + Sync {
    async fn enumerate_outputs(&self) -> Result<Vec<OutputInfo>>;
    async fn start_capture(&mut self, output: OutputId, config: CaptureConfig) -> Result<FrameReceiver>;
    async fn stop_capture(&mut self, output: OutputId) -> Result<()>;
}
```

### Выбор бэкенда

| Платформа | Основной бэкенд | Резервный |
|-----------|----------------|----------|
| Windows   | DXGI Desktop Duplication | GDI BitBlt |
| macOS     | ScreenCaptureKit (SCStream) | CGWindowListCreateImage |
| Linux     | PipeWire (через xdg-desktop-portal) | XShm / XFixes |

### Жизненный цикл кадра

1. `FrameProvider::start_capture` возвращает `FrameReceiver` — ограниченный MPSC-канал, передающий структуры `Frame`
2. Каждый `Frame` владеет буфером разделяемой памяти (`Arc<FrameBuffer>`), который при наличии ссылается на память GPU
3. Кодировщик WebRTC потребляет из канала; когда все ссылки `Arc` освобождаются, буфер возвращается в пул повторного использования
4. Поток захвата никогда не блокируется на кодировании — если канал заполнен, самый старый кадр отбрасывается, и счётчик `FrameDropped` увеличивается

### Цветовое пространство и формат

- Все бэкенды согласовывают формат с наибольшей доступной битовой глубиной (BGRA8, NV12 или P010)
- Шаг `ColorSpaceConverter` выполняет преобразование в предпочтительный формат кодировщика
- Метаданные HDR сохраняются, если источник их предоставляет

## Последствия

### Положительные

- Чёткое разделение захвата и кодирования позволяет независимое тестирование
- Путь zero-copy на Windows (DXGI) и macOS (ScreenCaptureKit) удерживает задержку в пределах бюджета
- Дизайн на основе типажа допускает сторонние бэкенды (например, виртуальные дисплеи, тестовые источники) без изменения основного кода
- Пул буферов кадров снижает нагрузку выделения памяти при продолжительном захвате

### Отрицательные

- PipeWire на Linux добавляет зависимость от D-Bus, что усложняет сценарии headless/встраиваемых систем
- Разрешение на запись экрана macOS требует взаимодействия с пользователем при первом запуске — без обходного пути
- Поддержка четырёх реализаций бэкенда увеличивает поверхность тестирования

### Риски и смягчения

- **Риск**: Изменения API PipeWire между дистрибутивами. **Смягчение**: зафиксировать стабильный C API `pw_stream` и вендорить Rust-привязки.
- **Риск**: Прерывистость DXGI на ноутбуках с гибридным GPU. **Смягчение**: определять топологию GPU при запуске и предпочитать интегрированный GPU для захвата, когда дискретный GPU выполняет рендеринг.
