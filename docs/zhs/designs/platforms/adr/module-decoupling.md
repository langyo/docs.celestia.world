# 模块解耦 -- 每模块 types.rs

- **状态**：已接受
- **日期**：2026-06-09
- **作者**：Evernight 核心团队

## 背景

Evernight 早期将所有公共类型（SSH 相关、WebRTC 相关、协议消息等）集中在单一的 `types.rs` 文件中。随着模块增多，该文件膨胀严重，导致无关模块间产生隐式耦合，修改一个模块的类型常常引发其他模块的重新编译。

关键约束：

- 各模块须可独立编译和测试
- 模块间通信需要共享的消息/协议类型
- 公共类型变更不应触发无关模块的重新编译
- Cargo feature gates 要求类型定义与模块边界对齐

## 决策

采用**每模块独立 `types.rs`** 的架构。具体策略：

- 每个功能模块（screen、ssh、webrtc、tunnel、hardware）维护各自的 `types.rs`
- 模块内部类型仅在模块 `types.rs` 中定义和导出
- 跨模块共享的协议类型放在 `protocol/types.rs` 中
- 模块间通过 `protocol` 模块定义的消息结构进行通信，不直接依赖彼此的内部类型
- 各模块的 `types.rs` 仅导出该模块公共接口所需的最小类型集合

## 影响

### 积极影响

- 模块间耦合度大幅降低，各模块可独立开发和测试
- 类型变更的编译影响范围缩小至相关模块
- 清晰的类型所有权边界，新贡献者更容易定位类型定义
- 与 Cargo feature gates 天然对齐，未启用的模块类型不参与编译

### 消极影响

- 跨模块类型转换可能需要额外的 `From` 实现
- 协议层类型需要与各模块内部类型保持同步，增加维护成本
