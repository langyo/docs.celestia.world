# 기여 가이드 (Entelecheia)

> [`CONTRIBUTING.md`](../meta/CONTRIBUTING.md)를 참조하십시오. 명령 자체는 번역하지 않습니다. 모호한 점이 있으면 영문 버전을 기준으로 합니다.

## 기여 정책 (먼저 읽어주십시오)

Entelecheia는 물리 및 산업 시스템을 구동할 수 있으므로, **안정성과 보안이 기여 처리량보다 우선**됩니다. Pull Request를 제출하기 전에 본 절을 먼저 읽어주십시오.

- **병합 기준이 높으며, 공개 로드맵이 아닙니다.** PR 제출이 곧 병합을 의미하지는 않습니다. 의도적으로 수량을 적게 유지하며, 아키텍처에 부합하고 검토를 통과한 변경만 수용합니다. 이는 의도적인 정책이며 무례함이 아닙니다.
- **환영하는 기여:** 버그 보고, 집중된 수정, **외연**(Layer 3 플러그인, 장치 profile, LLM

provider 어댑터, 통합, 문서)에 대한 범위가 명확한 개선, 그리고 코드 작성 전 설계 논의.

- **일반적으로 병합되지 않는 것:** 대규모 사전 제안 없는 재작성, 사전 설계 논의 없는 아키텍처 변경, 대량의 "vibe-coded"

PR, 핵심 보안 또는 정확성 기준을 낮추는 모든 변경, 그리고 초대 및 장기 검토 없이 보안이 중요한 핵심에 대한 변경.

- **핵심 vs. 외연.** 핵심(오케스트레이션, 마이크로커널, 보안)은 가장 엄격한 기준을 유지하며 주로 핵심 팀이 유지 관리합니다. 외연은

외부 기여가 가장 유용하고 수용될 가능성이 가장 높은 곳입니다.

- **CLA 서명 필수.** 모든 수용된 기여는 기여자 라이선스 동의서 서명이 필요합니다. [`CLA.md`](../meta/cla.md)를 참조하십시오.

제출 시 `Signed-off-by` (`git commit -s`)가 포함되어야 합니다.

> **라이선스는 개방되지만, 병합 기준은 그렇지 않습니다.** **2030-01-01**에 본 프로젝트는 BUSL-1.1에서 SySL-1.0(수신자 선택)으로 전환됩니다. [`LICENSE`](../../../LICENSE)를 참조하십시오. 이는 *코드로 할 수 있는 일*을 완화할 뿐,
> 검토 기준을 낮추거나 CLA를 취소하거나 더 많은 PR을 수용함을 의미하지 **않습니다**. 변경일 전후로 기여 정책은 변하지 않습니다.

## 보안

공개 issue로 보안 취약점을 보고하지 **마십시오**.
[GitHub Security Advisories](https://github.com/celestia-island/entelecheia/security/advisories/new)를 통해 비공개로 보고하십시오. 위협 모델 및 대응 SLA는 [`SECURITY.md`](../meta/security.md)를 참조하십시오.

## 행동 강령

존중, 건설적, 포용적인 태도를 유지해 주십시오. [Contributor Covenant Code of Conduct](../meta/code-of-conduct.md)를 따릅니다.

## Pull Request 절차

1. Fork하고 `main`에서 브랜치를 생성하십시오.
1. 큰 변경은 먼저 issue에서 논의하십시오.
1. 커밋은 원자적(atomic)으로, Conventional Commits를 따르십시오.
1. `just ci`(또는 저장소의 CI 명령)가 통과하는지 확인하십시오.
1. CLA에 서명하고 `Signed-off-by`를 추가하십시오.
1. 리뷰 의견에 답변하십시오. force-push는 rebase 용도로만 사용하십시오.

## 라이선스 및 CLA

**BUSL-1.1**을 채택하며, **변경일은 2030-01-01**로, 이후 수신자 선택의 **SySL-1.0**으로 전환됩니다. 현재 내부 운영, 학술, 정부, 교육 및 비상업적 용도에서는 이미 SySL-1.0과 동등합니다([`LICENSE`](../../../LICENSE)의 Additional Use Grant 참조). 제한된 상업적 사용(호스팅, 재판매, 서비스로의 리브랜딩 판매)은 변경일 이전에 별도의 상업 라이선스를 취득해야 합니다.

기여 제출은 본 프로젝트의 라이선스에 따라 라이선스가 부여됨에 동의하고 CLA([`CLA.md`](../meta/cla.md))에 서명함을 의미합니다. CLA는
프로젝트에 **재라이선스 권한을 포함한** 관대한 라이선스를 부여하여, 프로젝트가 BUSL→SySL 경로를 유지하고 향후 라이선스를 조정할 수 있도록 합니다.
