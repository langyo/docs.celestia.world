## 貢献ポリシー（最初にお読みください）

Entelecheia は物理システムおよび産業システムを駆動できるため、**安定性とセキュリティが貢献スループットよりも優先されます**。Pull Request を提出する前に、本セクションをお読みください。

- **マージの敷居は高く、公開ロードマップではありません。** PR を提出してもマージされるとは限りません。意図的に少数に絞られ、アーキテクチャに適合しレビューを通過した変更のみを受け入れます。これは意図的なものであり、無礼ではありません。
- **歓迎される貢献：** バグ報告、焦点を絞った修正、**外延**（Layer 3 プラグイン、デバイスプロファイル、LLM provider アダプター、統合、ドキュメント）に対する範囲の明確な改善、およびコードを書く前の設計議論。
- **通常マージされないもの：** 提案なしの大規模な書き換え、事前の設計議論なしのアーキテクチャ変更、大量の「vibe-coded」PR、コアのセキュリティや正当性の基準を下げるあらゆる変更、および招待なしで長期レビューを要するセキュリティクリティカルなコアへの変更。
- **コア vs. 外延。** コア（オーケストレーション、マイクロカーネル、セキュリティ）は最も厳格な基準を維持し、主にコアチームが保守します。外延は外部貢献が最も有用で、受け入れられる可能性が最も高い場所です。
- **CLA への署名が必須です。** 受け入れられるすべての貢献には貢献者ライセンス同意書への署名が必要です。[`CLA.md`](../meta/cla.md) を参照してください。提出には `Signed-off-by`（`git commit -s`）が必要です。

> **ライセンスは開放されますが、マージの敷居は変わりません。** **2030-01-01** に、本プロジェクトは BUSL-1.1 から SySL-1.0（受領者選択）に移行します。[`LICENSE`](../../../LICENSE) を参照してください。これは*コードで何ができるか*を緩和するものであり、**レビュー基準を下げたり、CLA を撤廃したり、より多くの PR を受け入れることを意味するものではありません**。変更日前後で貢献ポリシーは変わりません。

## セキュリティ

公開 issue でセキュリティ脆弱性を報告**しないでください**。
[GitHub Security Advisories](https://github.com/celestia-island/entelecheia/security/advisories/new)
を通じて非公開で報告してください。脅威モデルと対応 SLA については [`SECURITY.md`](../meta/security.md) を参照してください。

## 行動規範

敬意を持って、建設的で、包摂的であり続けてください。私たちは [Contributor Covenant Code of Conduct](../meta/code-of-conduct.md) に従います。

## Pull Request フロー

1. Fork して `main` からブランチを作成。
1. 大きな変更は事前に issue で議論。
1. コミットはアトミックに、Conventional Commits に従う。
1. `just ci`（またはリポジトリの CI コマンド）が通過することを確認。
1. CLA に署名し `Signed-off-by` を追加。
1. レビューコメントに返信。force-push は rebase の場合のみ使用。

## ライセンスと CLA

**BUSL-1.1** を採用し、**Change Date は 2030-01-01**、その時点で受領者選択の **SySL-1.0** に移行します。現在、内部運用、学術、政府、教育、非商用利用においては、既に SySL-1.0 と同等です（[`LICENSE`](../../../LICENSE) の Additional Use Grant を参照）。制限付きの商用利用（ホスティング、転売、サービスとしての再販）は、Change Date 前に別途商用ライセンスが必要です。

貢献を提出することにより、あなたの貢献が本プロジェクトのライセンスに基づいてライセンスされることに同意し、CLA（[`CLA.md`](../meta/cla.md)）に署名したものとみなされます。CLA はプロジェクトに**サブライセンス権を含む**寛容なライセンスを付与し、プロジェクトが BUSL→SySL のパスを維持し、将来ライセンスを調整できるようにします。
